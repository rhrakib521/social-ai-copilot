Architecture Approaches

Three ways to structure the extension internals. Pick the one that fits your vision.

A

Flat Module Architecture (Recommended)

Each file is a self-contained ES module with a clear single responsibility. content.js orchestrates DOM work, background.js handles API calls and state, core/ modules are imported as needed. No classes, no inheritance — just functions and exports.

Pros

  • Simple to understand — each file does one thing
  • Easy to debug — call stacks are shallow
  • Matches the spec's file structure directly
  • Fastest to build

Cons

  • Can grow unwieldy if features balloon
  • Shared state management is ad-hoc
B

Event-Driven Architecture

A central event bus connects all modules. Content scripts fire events like field-detected, ai-requested, text-generated. Background worker listens and dispatches. Modules communicate only through events — no direct imports between content and background.

Pros

  • Very decoupled — modules don't know about each other
  • Easy to add new platforms or providers
  • Natural fit for Chrome's message-passing model

Cons

  • Harder to trace execution flow
  • Event naming becomes an implicit API
  • More boilerplate for simple operations
C

Class-Based OOP

Each subsystem is a class: AIProviderManager, PlatformDetector, UIManager, etc. A central CopilotApp class wires them together. Inheritance for providers (base AIProvider → OpenAIProvider).

Pros

  • Clear encapsulation
  • Type-safe patterns (if adding TypeScript later)
  • Familiar to OOP-background developers

Cons

  • Overkill for an extension this size
  • this binding issues in content scripts
  • Harder to tree-shake / lazy-load
